# MonoGame.Framework.Facade

A tiny "facade" assembly that satisfies SDV's compile-time reference to
`MonoGame.Framework` v3.8.0.1641 by **forwarding** every public type to the
corresponding KNI assembly (`Xna.Framework`, `Xna.Framework.Game`, etc.).

## Why this exists

`Stardew Valley.dll` (the GOG release, v1.6.x) was compiled against the
upstream `MonoGame.Framework` assembly. It carries this AssemblyRef:

```
MonoGame.Framework, Version=3.8.0.1641, Culture=neutral, PublicKeyToken=null
```

Our runtime uses [KNI](https://github.com/AristurtleDev/kni) (a MonoGame fork
that targets Blazor WebAssembly via `nkast.Xna.Framework.*` packages), whose
assemblies are named `Xna.Framework`, `Xna.Framework.Game`, etc. — *not*
`MonoGame.Framework`. The CLR's type resolution rules forbid loading two
different assemblies with the same identity, so we can't just rename KNI's
output.

The standard .NET solution is **type forwarding**: ship a small assembly
named `MonoGame.Framework` (matching SDV's AssemblyRef) whose only metadata
is a `TypeForwardedTo` entry for each public type, pointing at the KNI
assembly that actually implements it. The CLR follows the forwarders
transparently — SDV code that references `Microsoft.Xna.Framework.Game`
resolves the type from `MonoGame.Framework` (the facade), then the runtime
follows the forwarder to `Xna.Framework.Game` (KNI) where the type's IL
actually lives.

## How it's built

`AssemblyInfo.cs` is generated by `scripts/generate-facade-types.sh`, which:

1. Loads the 5 KNI assemblies (`Xna.Framework`, `.Game`, `.Graphics`,
   `.Content`, `.Input`) into a small throwaway enumerator process.
2. Walks `Assembly.GetTypes()` on each.
3. Filters to public, non-generic, non-open-generic types.
4. Emits one `[assembly: TypeForwardedTo(typeof(Full.Type.Name))]` line per
   type, plus `[assembly: AssemblyVersion("3.8.0.1641")]` to pin the version
   to what SDV expects.

Re-run the generator after KNI upgrades to pick up newly-added types:

```bash
./scripts/generate-facade-types.sh
dotnet build src/MonoGame.Framework.Facade
```

## What the facade does NOT do

- It contains **no IL of its own** — no methods, no fields, no static
  constructors. It's pure metadata.
- It does **not** shim or wrap anything. Type lookups go straight to KNI.
- It does **not** patch the SDV DLL. The SDV DLL is loaded byte-for-byte
  unmodified.
- It does **not** handle SMAPI. SMAPI has its own AssemblyRefs and needs
  its own facade strategy (Phase 3).

## Constraints

- The facade assembly must be loaded into the same `AssemblyLoadContext`
  as the SDV DLL, OR be discoverable by the ALC's `Resolving` callback.
  See `SdvWebPort.PoC.SdvLoad/Program.cs` for the canonical pattern.
- If a future KNI version moves a type between assemblies (e.g., `Color`
  moved from `Xna.Framework` to `Xna.Framework.Graphics`), the generator
  picks up the new location automatically — no manual edits needed.
- Strong-name signing: SDV 1.6 ships against unsigned MonoGame.Framework.
  If a future SDV release references a signed MonoGame.Framework, the
  facade would need to be signed with the matching key (which we don't
  have). Workaround: enable strong-name bypass for the ALC, or fork the
  facade and have the user re-sign it locally.

## Output

- `bin/Debug/net10.0/MonoGame.Framework.dll` — ~14KB
- 337 forwarded types (as of KNI v4.2.9001):
  - 27 from `Xna.Framework` (Vector2, Matrix, BoundingBox, etc.)
  - 19 from `Xna.Framework.Game` (Game, GameComponent, GameServiceContainer, etc.)
  - 210 from `Xna.Framework.Graphics` (Texture2D, SpriteBatch, RenderTarget2D, etc.)
  - 18 from `Xna.Framework.Content` (ContentManager, ContentTypeReader, etc.)
  - 63 from `Xna.Framework.Input` (KeyboardState, MouseState, GamePadState, etc.)
